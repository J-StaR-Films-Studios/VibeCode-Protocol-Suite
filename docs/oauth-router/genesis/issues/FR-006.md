# [FR-006] OAuth Implementation Reuse and Limits

## Labels
`MUS`, `oauth`, `architecture`

## User Story
As a Pi extension author, I want to leverage Pi’s built-in OAuth primitives safely, so that I minimize duplicated auth logic while still supporting multiple accounts.

## Proposed Solution

### Overview
Reuse OAuth adapters from `@mariozechner/pi-ai/oauth` for account login and refresh, while storing each account independently inside the extension-owned store.

### Implementation Flow
1. Investigate Pi’s built-in OAuth provider exports
2. Reuse supported adapters such as `openai-codex`
3. Document why Pi’s built-in `/login` cannot solve multi-account storage for one provider entry

### Technical Approach

```ts
import { getOAuthProvider } from "@mariozechner/pi-ai/oauth";
const provider = getOAuthProvider("openai-codex");
```

### Key Considerations
- Pi’s default auth model is one credential per provider entry
- Reuse is valid for login/refresh primitives, not for multi-account persistence
- Generic OpenAI-compatible OAuth still requires upstream-specific adapters

## Acceptance Criteria

- [ ] The implementation reuses a supported Pi OAuth adapter where possible
- [ ] The extension stores multiple accounts independently of Pi’s auth slot
- [ ] The README documents the reuse boundary and remaining limitations
