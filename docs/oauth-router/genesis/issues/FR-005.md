# [FR-005] Safe Failover

## Labels
`MUS`, `failover`, `streaming`

## User Story
As a Pi user, I want automatic failover when it is still correct to retry, so that I get better reliability without corrupted partial responses.

## Proposed Solution

### Overview
Buffer non-meaningful early stream events and retry another account only before meaningful output is emitted.

### Implementation Flow
1. Start a request against the selected account
2. Buffer early structural events
3. Retry another eligible account only if the failure happens before meaningful output

### Technical Approach

```ts
if (!emittedMeaningfulOutput && failure) {
  retryAnotherAccount();
}
```

### Key Considerations
- `start` or empty block-start events do not count as meaningful output
- Any visible delta or completed content block ends safe failover eligibility
- Mid-stream switching must remain disabled unless correctness is guaranteed

## Acceptance Criteria

- [ ] Pre-output failures can retry another account
- [ ] Partial output does not trigger unsafe account switching
- [ ] Final errors report useful routing context when failover is exhausted
