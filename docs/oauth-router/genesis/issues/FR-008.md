# [FR-008] Pluggable Upstream OAuth Adapters and UI Indicators

## Labels
`Future`, `oauth`, `ux`

## User Story
As a Pi power user, I want pluggable upstream adapters and inline UI signals, so that I can route more providers with less manual config.

## Proposed Solution

### Overview
Add more upstream-specific OAuth adapters, richer per-model routing config, and optional inline UI notifications for active account and failover events.

### Implementation Flow
1. Define an adapter interface for new upstream OAuth types
2. Add UI affordances for active-account or failover hints
3. Expand setup docs with adapter examples

### Technical Approach

```ts
interface RouterOAuthAdapter {
  id: string;
  login(): Promise<OAuthCredentials>;
  refreshToken(): Promise<OAuthCredentials>;
}
```

### Key Considerations
- Keep adapter boundaries explicit
- Avoid over-coupling UI behavior to routing internals
- Preserve the no-daemon architecture

## Acceptance Criteria

- [ ] Adapter extension points are documented
- [ ] Future UI indicator approach is defined
- [ ] The roadmap identifies which upstreams should be added next
